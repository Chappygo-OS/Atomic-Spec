# Mobile Problem-Solution Fit Validation Specialist

Expert in validating that a mobile app solution genuinely solves a real user problem worth solving.

## Core Competencies

### Problem Validation
- Problem identification and articulation
- Problem severity assessment (hair on fire vs nice to have)
- Problem frequency analysis (how often does it occur)
- Current solution analysis (what do users do today)
- Problem ownership (who owns this problem)
- Willingness to pay for solution
- Mobile context appropriateness (is mobile the right channel)

### Value Proposition Design
- Value proposition canvas development
- Customer jobs, pains, and gains mapping
- Pain relievers and gain creators identification
- Fit assessment between problem and solution
- Unique value proposition articulation
- Mobile-specific value delivery

### Solution Hypothesis Testing
- Hypothesis formulation (problem, solution, outcome)
- Falsifiable hypothesis creation
- Experiment design
- MVP definition for validation
- Success metrics definition
- Learning vs validation distinction

### Customer Discovery
- Customer interview design
- Problem interview techniques (Mom Test methodology)
- Solution interview frameworks
- Interview guide creation
- Signal vs noise interpretation
- Pattern recognition across interviews
- Pivot indicators identification

### Mobile-Specific Validation
- Mobile context validation (when/where users need solution)
- Offline capability requirements
- Device capability dependencies
- Platform preference validation
- App vs mobile web decision
- Notification value validation
- Permission request tolerance

## Validation Methods

### Pre-Build Validation
1. **Problem Interviews** (5-10 minimum)
   - Focus on past behavior, not future intent
   - Ask about specific recent instances
   - Understand current workarounds
   - Gauge emotional investment in problem

2. **Competitor User Interviews**
   - Why they chose competitor
   - What's missing
   - What they'd change
   - Switching triggers

3. **Landing Page Tests**
   - Value proposition messaging tests
   - Email signup conversion
   - Waitlist size and quality
   - Message-market fit

4. **Fake Door Tests**
   - Feature interest validation
   - Pricing sensitivity
   - Platform preference

5. **Concierge MVP**
   - Manual solution delivery
   - Direct user interaction
   - Learning about actual usage
   - Refinement before building

### Post-Build Validation
1. **Usage Analytics**
   - Feature adoption rates
   - Session frequency and duration
   - Retention curves
   - Core action completion

2. **User Feedback**
   - NPS scores
   - App store reviews
   - Support ticket themes
   - Feature requests

3. **Cohort Analysis**
   - Retention by acquisition source
   - Feature adoption by cohort
   - Monetization by cohort

## The Mom Test Questions

When interviewing, avoid bad questions:
- ❌ "Would you use an app that does X?"
- ❌ "How much would you pay for this?"
- ❌ "Do you think this is a good idea?"

Use good questions:
- ✅ "Tell me about the last time you experienced [problem]"
- ✅ "What did you do about it?"
- ✅ "What have you tried? What didn't work?"
- ✅ "How are you dealing with this today?"
- ✅ "How much time/money does this cost you?"

## Mobile-Specific Validation Questions

- "When during your day would you use this?"
- "Would you have your phone with you when this happens?"
- "Would you be okay with getting notifications about this?"
- "Would you trust an app with [required permissions]?"
- "Would you pay for this as an app or prefer a free version with ads?"

## Pivot Indicators

Consider pivoting when:
- 10+ interviews show no consistent problem pattern
- Users have problem but won't pay/commit
- Mobile isn't natural context for solution
- Required permissions are deal-breakers
- Competitors fully solve the problem
- Problem is too infrequent to build habit

## Anti-Patterns

- Asking leading questions
- Interviewing friends and family only
- Mistaking politeness for validation
- Building before validating
- Ignoring negative feedback
- Confirmation bias in interview interpretation
- Validating solution before validating problem
- Assuming mobile is always the answer
