## Test Planning Framework

### Research Questions
```
Types of Questions:
- Exploratory: "How do users approach [task]?"
- Comparative: "Which design performs better?"
- Validation: "Can users complete [task] successfully?"
- Benchmark: "How does performance compare to baseline?"

Example Questions:
- Can users complete checkout in under 2 minutes?
- Do users find the search feature?
- Which onboarding flow has better completion?
- What causes users to abandon the cart?
```

### Test Methods Selection
```
Method by Goal:

Finding Usability Issues:
- Moderated testing (best for depth)
- Think-aloud protocol
- 5-8 participants per round

Measuring Performance:
- Unmoderated testing (larger sample)
- Task completion metrics
- 20+ participants for statistics

Comparing Designs:
- A/B testing
- Preference testing
- Within-subjects or between-subjects design

Quick Validation:
- Guerrilla testing
- First-click testing
- 5-second testing
```

### Participant Recruitment
```
Criteria to Define:
- Demographics (age, location, etc.)
- Experience level (novice, expert)
- Current behavior (existing users, competitors)
- Device usage (iOS, Android, tablet)
- Specific characteristics (frequent shoppers, etc.)

Recruitment Sources:
- User database
- Recruitment agencies
- Social media
- Intercept (guerrilla)
- Remote platforms (UserTesting, etc.)

Sample Sizes:
- Qualitative (finding issues): 5-8 per segment
- Quantitative (measuring): 20-30 minimum
- A/B testing: Statistical power calculation
```

### Screening Questions
```
Example Screener:
1. Which smartphone do you use? [iOS/Android/Other]
2. How often do you [relevant behavior]?
3. Have you used [app category] apps before?
4. When did you last [relevant action]?
5. How comfortable are you with technology? [Scale]

Disqualification:
- Work in UX/design/market research
- Work for competitors
- Participated in research recently
- Don't match target criteria
```

## Task Design

### Task Writing Guidelines
```
Good Task Characteristics:
- Realistic and goal-oriented
- Clear success criteria
- Appropriate complexity
- No leading language
- Single focus per task

Example Good Task:
"You want to order dinner for delivery tonight.
Find a restaurant that delivers pizza to your
location and add a large pepperoni pizza to
your order."

Avoid:
- "Use the search feature to find..."
- "Click on the menu icon..."
- "Easily complete the checkout..."
```

### Task Structure
```
Task Components:
1. Context/Scenario: Why are you doing this?
2. Goal: What do you want to accomplish?
3. Success criteria: How do we know it's done?
4. Time estimate: How long should it take?

Template:
"Imagine [context]. You want to [goal].
[Additional details if needed]."

Success Criteria Types:
- Binary: Completed or not
- Partial: Multiple components
- Time-based: Within X minutes
- Quality: Specific result achieved
```

### Mobile-Specific Tasks
```
Consider:
- Touch interactions
- Gesture-based actions
- One-handed scenarios
- Interruption recovery
- Offline functionality
- Permission flows
- Notification interactions

Example Mobile Tasks:
- "While walking, quickly check your account balance"
- "You're in a loud environment - place an order"
- "Your phone just received a notification from the app"
- "You're holding a coffee - complete this with one hand"
```

## Testing Protocols

### Moderated In-Person Protocol
```
Setup (5 min):
- Welcome participant
- Explain session format
- Sign consent form
- Confirm recording permission

Introduction (5 min):
- Brief interview (background, habits)
- Explain think-aloud protocol
- Practice think-aloud

Tasks (30-40 min):
- Present tasks one at a time
- Observe and note
- Ask follow-up questions
- Probe on interesting behaviors

Debrief (10 min):
- Overall impressions
- Specific questions
- Comparison with expectations
- Thank and compensate
```

### Moderated Remote Protocol
```
Pre-Session:
- Send joining instructions
- Test technology
- Prepare backup plan

Technology:
- Video conferencing (Zoom, Teams)
- Screen sharing (built-in or tool)
- Recording permission
- Backup device contact

Adaptations:
- Clear verbal instructions
- Screen share for task cards
- Ask for screen orientation
- Describe actions verbally
```

### Unmoderated Remote Protocol
```
Platform Setup:
- UserTesting, Maze, Lookback, etc.
- Prototype or live app
- Task instructions in platform
- Follow-up questions programmed

Task Instructions:
- Self-explanatory
- No facilitator to clarify
- Simpler than moderated
- Clear start and end points

Quality Control:
- Screening questions
- Attention checks
- Review submissions
- Remove invalid responses
```

### Guerrilla Testing Protocol
```
Preparation:
- Portable device (tablet/phone)
- Brief task list (2-3 tasks)
- Quick consent process
- Small incentive (coffee, candy)

Location Selection:
- Coffee shops
- Public spaces
- Target user locations
- Events and conferences

Execution:
- Brief approach ("5 minutes of your time?")
- Quick context ("testing a new app")
- Observe 1-2 tasks
- Quick follow-up question
- Thank and compensate
```

## Mobile Testing Considerations

### Device and Environment
```
Device Variables:
- iOS vs Android
- Phone vs tablet
- Screen size range
- OS version
- Device performance

Environment Variables:
- Indoor (controlled)
- Outdoor (realistic)
- Lighting conditions
- Noise levels
- Network conditions

Testing Setup:
- Participant's own device (realistic)
- Test device (controlled)
- Screen recording method
- Camera for hand/face
```

### Capturing Behavior
```
What to Capture:
- Screen activity
- Touch gestures
- Face/expressions
- Verbal commentary
- Environment context

Tools:
- Built-in screen recording
- Lookback, UserTesting
- External camera
- Eye tracking (lab)
```

### Mobile-Specific Observations
```
Watch For:
- Thumb reach and grip
- Two-handed vs one-handed
- Gesture discovery
- Touch precision issues
- Scroll vs tap confusion
- Back button/gesture usage
- Keyboard interaction
- Permission responses
- Notification handling
```

## Metrics and Measurement

### Task Metrics
```
Quantitative:
- Success rate (binary, partial)
- Time on task
- Error count
- Error type
- Tap count / interaction count
- Help requests
- Task abandonment

Qualitative:
- Ease rating (1-7 scale)
- Confidence level
- Satisfaction rating
- Think-aloud insights
```

### Session Metrics
```
System Usability Scale (SUS):
- 10 standardized questions
- Score 0-100
- Benchmark comparisons

Net Promoter Score (NPS):
- Likelihood to recommend
- Promoters, passives, detractors

Custom Satisfaction:
- Overall satisfaction
- Specific feature ratings
- Open-ended feedback
```

### Success Criteria
```
Benchmarks:
- Industry standards
- Competitor performance
- Previous version
- Stated goals

Example Targets:
- 90% task completion rate
- Average task time < 2 minutes
- SUS score > 68 (above average)
- 0 critical severity issues
```

## Analysis and Reporting

### Issue Documentation
```
Issue Template:
- ID: UST-001
- Screen/Flow: [Location]
- Description: [What happened]
- Frequency: [X of N participants]
- Severity: [Critical/High/Medium/Low]
- Evidence: [Quotes, timestamps]
- Recommendation: [Fix suggestion]
```

### Severity Scale
```
Critical (Blocker):
- Prevents task completion
- Data loss
- Major confusion
- Fix: Must fix before launch

High:
- Significant delays
- Many users affected
- Major frustration
- Fix: Should fix before launch

Medium:
- Minor delays
- Some users affected
- Noticeable friction
- Fix: Fix if time permits

Low:
- Cosmetic issues
- Minor inconvenience
- Few users affected
- Fix: Consider for future
```

### Report Structure
```
Executive Summary:
- Key findings (3-5 points)
- Overall assessment
- Critical recommendations

Methodology:
- Test objectives
- Participants
- Tasks and scenarios
- Metrics collected

Findings:
- Per-task results
- Issues identified
- Patterns observed
- Positive findings

Recommendations:
- Prioritized fix list
- Quick wins
- Strategic improvements
- Next steps
```

### Analysis Tools
- Affinity mapping (Miro, FigJam)
- Rainbow spreadsheets
- Video annotation tools
- Statistical analysis (for quant)

## Anti-Patterns

- Testing with colleagues instead of real users
- Leading participants toward desired behavior
- Too many tasks in one session
- Not testing on actual mobile devices
- Ignoring environmental context
- Testing only happy paths
- Fixing issues without retesting
- Report without actionable recommendations
- Only testing at the end of design
- Participant pool too homogeneous
- No baseline for comparison
- Confusing opinions with usability issues
